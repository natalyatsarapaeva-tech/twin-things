# План приложения «Twin Things» — совместный каталог домашних вещей

> Рабочее название: **Twin Things** (плейсхолдер, ТЗ-вопрос №5 — можно переименовать позже).
> Архитектура семейства Twin: статические HTML на GitHub Pages, **отдельный**
> Firebase-проект (Auth + Firestore + Storage), GPT-4o через Cloudflare Worker `/ai`.
>
> **Статус:** Wave 0 (каркас + доступ) заложен в этом репозитории. Разработка
> идёт в ветке `claude/app-planning-spec-e2rfeo`. Перед запуском — см. `SETUP.md`
> (создание Firebase-проекта, деплой правил, GitHub Pages).

---

## 0. Решения, принятые на старте (обновлено)

Из ТЗ + уточнения владельца:

1. **Отдельный Firebase-проект** — не подселять в `natas-kitchen`.
2. **Auth с первого дня** (Google + Email/Password), модель **много-ко-многим**:
   у пользователя несколько каталогов, у каталога несколько участников.
3. **Роли в MVP — только `owner` / `editor`.** `viewer` (только чтение,
   «бабушке показать») откладываем в отдельную волну. Это упрощает правила:
   в MVP «участник = может править», проверка роли нужна только для действий
   владельца (управление участниками, удаление каталога).
4. **Кроп по объектам** на фото при AI-разборе — **v2**. В MVP исходное фото
   прикрепляется ко всем распознанным вещам.
5. **Совместный доступ — с самого начала** (Wave 2).

## 1. Суть

Личный и совместный каталог домашнего имущества. Каждая вещь — карточка с фото,
категорией, тегами и характеристиками. Каталогов («баз») у пользователя может
быть несколько (свой дом, дача, каталог родителей), в любой можно пригласить
других. Ключевая фича — **массовое создание вещей ИИ** по фото или тексту/голосу:
ИИ распознаёт объекты и возвращает список карточек; пользователь правит чек-лист
и сохраняет.

Сценарии: семейный учёт (страховка, переезд, «что где лежит»), быстрая
инвентаризация шкафа по фото, «что продать/отдать», гарантии и вишлист,
совместное ведение с ролями, поиск («где зимние шины», тег «хрупкое»).

## 4. Многопользовательская модель (много-ко-многим)

Пользователь ↔ каталог — «многие-ко-многим». Отличие от kitchen (там
один-юзер→одна-семья): у пользователя **список каталогов** и **активный каталог**.

### 4.1 Роли (MVP: две)
- **owner** — всё + управление участниками + удаление каталога.
- **editor** — полный CRUD вещей/категорий/тегов/фото.
- ~~viewer~~ — только чтение; **отложено** (отдельная волна). Заклад под него в
  правилах оставляем комментарием, но не реализуем.

### 4.2 Приглашения
- **Код присоединения (`joinCode`)** на каталоге — основной способ в MVP (готово
  в kitchen: `makeJoinCode`/`normalizeJoinCode`, поиск по коду). Быстро для семьи.
- **Email-инвайт** (`catalogs/{cid}/invites/{id}` `{email,role,status:"pending"}`
  + `collectionGroup('invites')` по своему email) — Wave 2, поверх кода.
- Владелец убирает участника удалением `members/{uid}`.

### 4.3 Активный каталог
- Свитчер в хедере; выбор в `localStorage('activeCatalogId')`.
- При первом входе авто-создаётся личный каталог «Мой дом» (пользователь = owner).

## 5. Модель данных (Firestore + Storage)

Вещи лежат **подколлекцией внутри каталога** — изоляция данных, доступ = членство.

```
users/{uid}                              — профиль {displayName, email, createdAt}
users/{uid}/catalogs/{catalogId}         — ИНДЕКС «мои каталоги» (денормализация):
                                           {role, name, joinedAt} — для свитчера
catalogs/{catalogId}                     — {name, ownerUid, joinCode, createdAt}
catalogs/{catalogId}/members/{uid}       — ИСТОЧНИК ПРАВ: {role, addedBy, joinedAt}
catalogs/{catalogId}/invites/{inviteId}  — {email, role, invitedBy, status, createdAt}
catalogs/{catalogId}/items/{itemId}      — карточки вещей (см. 5.1)
catalogs/{catalogId}/meta/categories     — категории каталога
catalogs/{catalogId}/meta/tags           — теги каталога
catalogs/{catalogId}/meta/categoryTemplates — шаблоны характеристик по категориям
```

> Источник прав — `members/{uid}` (пишет только owner). `users/{uid}/catalogs/*` —
> лишь денормализованный индекс для быстрого списка; сам по себе доступа не даёт.
> При принятии инвайта/выходе пишутся/удаляются обе записи.

### 5.1 Документ вещи `catalogs/{cid}/items/{id}`
```js
{
  id: "item-<ts>-<rand>",
  name: "Дрель Bosch GSB 13",
  description: "string",          // может сгенерировать ИИ
  category: "electronics",        // ОДНА категория (id из meta/categories)
  tags: ["fragile", "for-sale"],  // сквозные метки (id из meta/tags)
  characteristics: [              // гибкий список пар — §5.2
    { label: "Бренд", value: "Bosch" },
    { label: "Цена",  value: "3500", type: "money" },
    { label: "Куплено", value: "2024-03-10", type: "date" }
  ],
  photos: [                       // §7
    { path:"catalogs/cid/items/item-x/1.jpg", url:"…", thumbUrl:"…",
      isPrimary:true, w:1600, h:1200, uploadedAt:"ISO" }
  ],
  location: "Гараж / полка 2",
  quantity: 1,
  status: "have",                 // have | wishlist | lent | discarded
  source: "manual",              // manual | ai-photo | ai-text
  aiGenerated: false,
  createdBy: "uid",               // атрибуция; права — по членству
  createdAt: "ISO", updatedAt: "ISO"
}
```

### 5.2 Характеристики — гибридная модель
- Хранятся как массив пар `{label, value, type?}` — полностью гибко.
- Категория задаёт **шаблон** (`meta/categoryTemplates`), подставляемый при
  создании и заполняемый ИИ:
  ```
  electronics: [Бренд, Модель, Серийный №, Гарантия до, Цена, Дата покупки]
  furniture:   [Материал, Цвет, Размеры, Комната]
  clothing:    [Размер, Бренд, Сезон, Цвет]
  books:       [Автор, Год, Язык]
  generic:     [Состояние, Цена, Где куплено, Количество]
  ```
- `type` управляет виджетом: `text | number | money | date | select`.

### 5.3 Категории vs теги (на уровне каталога)
- **Категория** — одна, основная; таксономия общая для каталога (единый набор у семьи).
- **Теги** — много, сквозные (хрупкое, на продажу, детское, сезонное).
- Обе живут в `catalogs/{cid}/meta/*`, правят owner/editor (паттерн редактора тегов Twin).

## 6. Безопасность (Firestore + Storage rules)

Доступ = членство. **Урок kitchen: не запускать в test mode.** MVP-роли owner/editor
(всё, что «write», доступно любому члену; owner-only — управление участниками/каталогом).

### Firestore (эскиз, MVP owner/editor)
```
match /databases/{db}/documents {
  function signedIn() { return request.auth != null; }
  function uid() { return request.auth.uid; }

  match /users/{u} {
    allow read, write: if signedIn() && u == uid();
    match /catalogs/{cid} { allow read, write: if signedIn() && u == uid(); } // индекс
  }

  match /catalogs/{cid} {
    function isMember() { return exists(/databases/$(db)/documents/catalogs/$(cid)/members/$(uid())); }
    function role()     { return get(/databases/$(db)/documents/catalogs/$(cid)/members/$(uid())).data.role; }

    allow read:   if isMember();
    allow create: if signedIn() && request.resource.data.ownerUid == uid();
    allow update, delete: if isMember() && role() == 'owner';

    match /members/{m} {
      allow read:  if isMember();
      // владелец рулит участниками; себя-первого пишет создатель каталога
      allow write: if isMember() && role() == 'owner';
    }
    match /invites/{i} {                       // Wave 2
      allow read:   if isMember() || resource.data.email == request.auth.token.email;
      allow create, delete: if isMember() && role() == 'owner'; // editor-инвайты — позже
    }
    match /items/{id} { allow read: if isMember(); allow write: if isMember(); } // owner|editor
    match /meta/{id}  { allow read: if isMember(); allow write: if isMember(); }
  }
}
```
> `get()/exists()` на `members` — на каждую операцию с вещью; при семейном
> масштабе приемлемо. На больших объёмах — кешировать роль в custom claims.

### Storage rules (доступ к файлам по членству)
```
match /catalogs/{cid}/{allPaths=**} {
  allow read, write: if request.auth != null
    && firestore.exists(/databases/(default)/documents/catalogs/$(cid)/members/$(request.auth.uid));
}
```

### Индексы
- `collectionGroup('invites')` по `email` — экран «входящие приглашения» (Wave 2).
- `collectionGroup('items')` — только если понадобится глобальный поиск по всем
  каталогам; иначе поиск в пределах активного каталога (индекс не нужен).

## 7. Хранение фотографий (ключевой новый компонент)

Firestore-документ ≤1 МБ, base64 нельзя → **Firebase Storage**.

**Пайплайн на клиенте** (переиспользуем canvas-код twin):
1. `<input type=file multiple>` — 1–N фото.
2. Сжатие через canvas: основное — макс. сторона ~1600px, JPEG q≈0.8; **превью** ~300px.
3. Загрузка в Storage: `catalogs/{cid}/items/{itemId}/{n}.jpg` (+ `{n}_thumb.jpg`).
4. В документ вещи: `{path, url, thumbUrl, isPrimary, w, h}`.

Замечания: грид — `thumbUrl`; детальная карточка — полноразмер (lazy). Первое фото
`isPrimary` (обложка), можно переназначить. При удалении вещи/каталога — чистить
файлы (сироты). EXIF-ориентация нормализуется на canvas. Кроп по объектам — v2.

## 8. Экраны (зеркалят Twin + экраны каталогов)

- **`index.html` — каталог (активный):** свитчер каталогов в хедере; грид карточек
  с превью, под фото название/категория-чип/теги; фильтры (категория/тег/статус/
  локация) + поиск + сортировка; хедер-статистика (всего, суммарная ценность, N на
  продажу); FAB (+ добавить, ✨ AI, ⚙ настройки).
- **`item.html` — карточка вещи:** галерея фото (свайп/сетка, добавить/удалить/
  обложка); название/описание/категория/теги; характеристики (пары с типами,
  «+ поле»); локация/количество/статус; атрибуция «добавил(а)…».
- **`add-item.html` — добавление:** ручная форма (название → шаблон характеристик
  по категории, теги, фото, локация) + AI bulk-импорт (вкладки «По фото» / «По тексту»).
- **`rooms.html` (аналог mindmap):** дерево локация/категория → вещи; быстрые
  действия в детальной панели (сменить статус без ухода со страницы).
- **`settings.html` (аналог context.html):** редактор категорий/тегов/шаблонов;
  управление каталогом (переименовать, участники+роли, приглашения, входящие
  инвайты, выйти/удалить); профиль пользователя.

## 9. AI-функциональность (главная фича)

Batch-паттерн Twin: **разбор → редактируемый чек-лист → сохранить выбранное.**
Все результаты пишутся в **активный каталог**. Системный промпт получает
актуальные категории/теги текущего каталога (из `meta/*`, не хардкод) и требует
«только JSON, категории/теги только из предложенных id».

- **8.1 Фото → вещи (vision):** 1–N фото → GPT-4o vision через `/ai` → JSON-массив
  `[{name,description,category,tags[],characteristics[]}]` → чек-лист (чекбоксы по
  объекту) → сохранение `source:"ai-photo"`, `aiGenerated:true`, исходные фото
  прикрепляются ко всем найденным (кроп по bbox — v2).
- **8.2 Текст/голос → вещи:** как bulk-import Twin — диктовка/вставка → тот же JSON → тот же чек-лист.
- **8.3 Одиночное авто-описание:** «✨ Описать по фото» на карточке — ИИ заполняет описание и предлагает характеристики/теги.

## 10. Этапы разработки (waves) — скорректировано под реальное переиспользование

**Wave 0 — каркас + доступ**
- Репозиторий `twin-things`, новый Firebase-проект (владелец создаёт вручную).
- Дизайн-система twin → `styles.css`; `voice.js`, `pwa.js`, `sw.js`, `manifest.json`.
- Auth (Google + Email/Password) — **адаптация `firebase.js`/`store.js` kitchen**.
- Модель `catalogs / members / users-index` (много-ко-многим: список + активный).
- `catalog-core.js` (чистый) + тесты `node --test`; правила Firestore + Storage.
- Свитчер активного каталога, авто-создание «Мой дом» при первом входе.

**Wave 1 — ядро каталога (MVP)**
- `index.html` (грид + фильтры + поиск), `item.html` (галерея, характеристики,
  теги), `add-item.html` (ручная форма), редактор категорий/тегов.
- **Пайплайн фото:** сжатие + превью + Storage (§7).

**Wave 2 — совместный доступ**
- `settings.html`: участники и роли (owner/editor), приглашения по email + joinCode,
  входящие приглашения, выход/удаление каталога.

**Wave 3 — AI-импорт (главная фича)**
- Фото → вещи (vision), текст/голос → вещи, редактируемый чек-лист; авто-описание.

**Wave 4 — организация и полировка**
- `rooms.html` (по локациям/категориям); суммарная ценность; экспорт CSV/PDF (страховка/переезд).

**Wave 5 — умные фичи (v2)**
- **`viewer`-роль** (только чтение — «бабушке показать»).
- **Кроп по объектам** (bbox от модели).
- Напоминания о гарантии/сезонных вещах (Web Push — воркер twin переиспользуем).
- Интейк «фото из Telegram/почты» (эндпоинты воркера).
